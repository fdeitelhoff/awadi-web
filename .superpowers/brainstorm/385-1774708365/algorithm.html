Section 2 of 3 — Algorithm Design

Seven steps from raw data to a draft tour. Does this flow make sense?

1

Data fetch

Load all wartungsvertraege where datum_naechste_wartung falls within the selected window. Join with anlagen to get GPS coordinates and techniker_id. Group plants by technician. Load all technician profiles (working hours, start coordinates) and their absences from user_abwesenheiten.

2

Capacity map

For every technician × every calendar day in the window, calculate available minutes:

  • Start with working hours for that weekday (e.g. Mo 08:00–17:00 = 540 min)
  • Subtract any absence overlap for that day (full day → 0 min; partial e.g. 08:00–12:00 → 300 min remain, route starts at 12:00)
  • If technician doesn't work that weekday → 0 min

Result: a map of available_minutes[technician][date] and route_start_time[technician][date].

3

Travel time matrix (Google Maps)

For each technician, collect all plant coordinates + start location. Check reisezeiten_cache first. For cache misses, call the Google Maps Distance Matrix API in batches of up to 25×25 pairs. Store all results back to cache. Output: a full N×N matrix of driving minutes between every pair of locations per technician.

4

Geographic clustering (k-means)

Per technician: set k = number of available working days in the window. Run k-means on the GPS coordinates of the technician's plants. Each cluster = one day's work. Assign clusters to days by proximity of cluster centroid to the technician's start location (closest cluster → first available day).

Capacity overflow: if a cluster's total maintenance time exceeds the available minutes for its day, move the farthest plant(s) to the next available day and re-check.

5

Route ordering within each day (Nearest Neighbor TSP)

For each technician's day cluster: start at the technician's start location, greedily pick the nearest unvisited plant (by drive time from the cached matrix), repeat until all plants visited. Then calculate arrival times by accumulating: route_start_time + Σ(fahrtzeit + dauer).

6

Absence redistribution

For plants that couldn't be scheduled because their technician is absent for all days in the window:

  1. Defer first: if the technician has any available days in the window (even if not all), push the plant to the nearest available day of the same technician.
  2. Redistribute if fully absent: find the geographically nearest available technician for each unscheduled plant. Insert the plant into that technician's best-fit day (fewest minutes remaining). Re-run Nearest Neighbor for that day. Set original_techniker_id so the dispatcher can see it's a coverage substitution.
7

Save as draft

Create one touren record with status = 'entwurf'. Create one tour_eintraege row per stop with calculated times, positions, and travel durations. Return the draft ID to the dispatcher UI for review and manual adjustment.

Tickets

Tickets are not included automatically by the algorithm — the dispatcher manually pulls them in from a sidebar after the draft is generated. When a ticket is added, it is inserted into the best-fit day (geographically closest cluster, enough remaining capacity) and Nearest Neighbor is re-run for that day only. Ticket status is set to 'eingeplant'.

Does this algorithm flow make sense? Any steps to adjust before Section 3 (UI)?
