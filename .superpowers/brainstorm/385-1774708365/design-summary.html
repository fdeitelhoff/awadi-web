Full Design Summary — Tour & Route Planning

All three sections. Approve to proceed to the spec document.

1 · Data Model

NEW

touren

Tour header: name, planning window (von/bis), status (entwurf/veröffentlicht), created_by

NEW

tour_eintraege

One row per stop: techniker_id, datum, position, item_type (wartung/ticket), anlage_id or ticket_id, arrival time, drive time, duration, original_techniker_id

NEW

reisezeiten_cache

Cached Google Maps travel times keyed by coordinate pairs. Avoids repeat API calls.

EXTEND

profiles

Add start_strasse/hausnr/plz/ort + start_lat/lng — configurable daily start location per technician

EXTEND

tickets

Add techniker_id (nullable). Status 'eingeplant' already exists for tour assignment.

NO CHANGE

abwesenheiten

von/bis are already timestamptz — supports full and partial-day absences today.

2 · Algorithm (Geographic Clustering + Nearest Neighbor)

1
Data fetch
Plants due in window + open tickets + technician profiles + absences
2
Capacity map
Available minutes per technician per day, accounting for absences (full or partial)
3
Travel matrix
Google Maps Distance Matrix, cache-first, batches of 25×25
4
Clustering
k-means per technician (k = available days). Capacity overflow → move farthest plant to next day
5
Route order
Nearest Neighbor TSP per day, starting from technician's start location
6
Absence cover
Defer within same tech first → then geo-nearest available tech. Flags original_techniker_id
7
Save draft
touren (entwurf) + all tour_eintraege with times and positions

Tickets are auto-included in the draft (opt-out model). Technician lookup: explicit techniker_id → anlage's techniker_id → geo-nearest. Dringend tickets prioritised.
