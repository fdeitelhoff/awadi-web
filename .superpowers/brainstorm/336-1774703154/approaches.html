Tour Planning — Algorithm Approaches

Three options for how the scheduling algorithm works. Pick the one that fits best.

A

Sequential Day Filling

How it works: Plants are sorted by location (nearby plants grouped together). Each available working day is filled with plants until the day's time capacity is full, then the next day begins. Within each day, stops are ordered using Nearest Neighbor from the technician's start point.


Pros

  • Simplest to implement
  • Fast execution
  • Easy to debug and explain
  • Lowest API cost (fewer Maps calls)

Cons

  • Can produce suboptimal routes
  • Geographic grouping is rough
  • May leave inefficient gaps on some days

Effort 2–3 days dev  |  Route quality ★★☆☆

B

Geographic Clustering + Nearest Neighbor ★ Recommended

How it works: Plants are first grouped geographically into clusters (one cluster per available working day) using k-means. Each cluster then becomes a day's work. Google Maps Distance Matrix is used to get real driving times between all plants, and Nearest Neighbor TSP orders the stops within each day starting from the technician's start address.


Pros

  • Much better route quality than A
  • Geographic clustering minimizes daily travel
  • Real driving times from Google Maps
  • Reasonable implementation complexity
  • Travel time cache in DB (no repeated API calls)

Cons

  • k-means can produce uneven clusters
  • Not globally optimal (still heuristic)
  • More Maps API calls upfront

Effort 4–5 days dev  |  Route quality ★★★☆

C

Clarke-Wright Savings Algorithm

How it works: A classic operations research algorithm for VRP. Starts with every plant having its own dedicated round-trip, then repeatedly merges routes where the "saving" (reduced travel) is greatest. Naturally handles multi-day scheduling and produces near-optimal results for the scale of a maintenance company.


Pros

  • Best route quality of the three
  • Industry standard for field service VRP
  • Handles capacity constraints well
  • Scales to larger plant counts

Cons

  • Most complex to implement
  • Harder to explain to users
  • Overkill for small plant counts (<30/technician)
  • Still needs Google Maps for travel times

Effort 6–8 days dev  |  Route quality ★★★★

My recommendation

Start with B (Geographic Clustering). It produces genuinely good routes, the implementation is manageable, and the dispatcher can fix the few cases where clusters aren't ideal via drag-and-drop anyway. Clarke-Wright is only worth the extra effort if you have technicians managing 50+ plants each and the dispatcher's corrections become too frequent.
